# PostFlow 完整功能测试方案

| 字段 | 内容 |
|------|------|
| 版本 | v1.0 |
| 日期 | 2026-07-05 |
| 适用项目 | `post-flow` 单仓 Next.js 全栈 MVP |
| 测试方式 | 本地 Agent + 本地浏览器 + 少量 API/数据库辅助验证 |
| 覆盖范围 | 用户端、商业化 Mock、管理端、鉴权、数据持久化、用户隔离、基础 UI/响应式 |

---

## 1. 测试目标

本测试方案用于验证 PostFlow 当前本地 MVP 是否具备“可注册、可登录、可持久化、可演示业务闭环”的完整可用性，并为本地 AI Agent 结合浏览器自动操作提供稳定测试脚本。

核心目标：

1. 验证用户可完成 `注册 -> 登录 -> 新建创作 -> AI 生成 -> 配图 -> 平台适配 -> 预览 -> 发布/定时 -> 日志/队列查看` 的完整闭环。
2. 验证 Auth.js 会话保护、API 鉴权、用户数据隔离和基础 IDOR 防护。
3. 验证商业化 Mock：个人主页、会员套餐、支付模拟、AI 额度加购、订单/付费数据展示。
4. 验证管理端 MVP：管理员权限、概览、用户、订单、任务中心。
5. 验证本地 Agent 在浏览器中可稳定执行回归测试，并输出清晰证据。

---

## 2. 当前实现边界

### 2.1 已实现并纳入测试

- Auth.js 邮箱密码注册、登录、退出和会话保护。
- Prisma + SQLite 持久化用户、草稿、版本、图片、平台版本、账号、发布任务、订单、支付记录。
- 用户端页面：Landing、登录、注册、Dashboard、Onboarding、Studio、Drafts、Images、Adapt、Preview、Publish、Queue、Accounts、Logs、Settings、Profile、Billing。
- 管理端页面：`/admin`、`/admin/users`、`/admin/users/[id]`、`/admin/orders`、`/admin/tasks`。
- API：`/api/auth/*`、`/api/v1/*`、`/api/admin/*`。
- DeepSeek / Seedream 未配置 Key 时的 Mock fallback。
- Mock 平台账号连接与 Mock 发布模拟。
- 商业化 Mock：套餐升级、额度加购、手工订单、支付标记。

### 2.2 当前不验证为真实能力

- 不验证真实小红书/微信公众号 OAuth。
- 不验证真实平台发布成功。
- 不验证真实支付扣款、真实发票和真实对账。
- 不验证生产对象存储；当前图片可本地 Data URL / Mock URL 演示。
- 不验证高并发和分布式任务队列；当前为本地 SQLite MVP。

## 3. 测试环境准备

### 3.1 基础命令

在仓库根目录执行：

```bash
cd /Users/admin/Documents/Github/post-flow
npm install
cp .env.example .env
npm run db:generate
npm run db:migrate
npm run dev
```

默认访问：

- 用户端：`http://localhost:3000`
- 管理端：`http://localhost:3000/admin`
- Prisma Studio：`npm run db:studio`

如果 `3000` 被占用，以终端输出端口为准，例如 `http://localhost:3001`。

### 3.2 建议 `.env`

```bash
DATABASE_URL="file:./dev.db"
AUTH_SECRET="local-dev-secret-change-me"
NEXTAUTH_URL="http://localhost:3000"
ADMIN_EMAILS="admin@postflow.local"
```

如果开发服务实际运行在 `3001`，建议同步改：

```bash
NEXTAUTH_URL="http://localhost:3001"
```

### 3.3 测试账号规划

| 账号 | 用途 | 密码 | 预期权限 |
|------|------|------|----------|
| `admin@postflow.local` | 管理员测试 | `Postflow123` | `isAdmin = true` |
| `creator-a-{timestamp}@postflow.local` | 普通用户 A | `Postflow123` | 普通用户 |
| `creator-b-{timestamp}@postflow.local` | 普通用户 B / 隔离测试 | `Postflow123` | 普通用户 |

注意：管理员账号需要在 `ADMIN_EMAILS` 中配置，并通过注册流程创建，才会自动获得管理权限。

### 3.4 本地 Agent 浏览器测试约定

Agent 执行浏览器测试时，统一采用以下协议：

1. 每个测试用例开始前说明当前 URL、账号、目标。
2. 优先通过可见文本、按钮名称、表单 label 定位元素。
3. 每步操作后观察页面是否有 loading、toast、跳转、错误提示。
4. 每个用例必须输出：`通过/失败`、实际结果、截图或关键页面文本、失败时的复现步骤。
5. 涉及数据持久化的用例，必须执行一次刷新或重新登录验证。
6. 涉及用户隔离的用例，必须切换到第二账号验证“看不到/改不了”。
7. 不在真实生产账号、真实支付账号、真实平台账号上执行本方案。

## 4. 测试分级与准入标准

| 等级 | 用途 | 必须通过条件 |
|------|------|--------------|
| P0 Smoke | 每次提交后快速验证 | 注册、登录、主流程发布、队列/日志、构建通过 |
| P0 Full | MVP 演示前完整验证 | 用户端闭环、管理端、商业化 Mock、隔离、安全全部通过 |
| P1 Regression | 大功能调整后回归 | 设置、响应式、错误提示、边界输入、视觉一致性 |
| P2 Exploratory | Agent 探索测试 | 随机路径、异常操作、长文本、重复点击、刷新恢复 |

Go / No-Go 标准：

- `npm run lint` 通过。
- `npm run build` 通过。
- P0 测试用例通过率 100%。
- P1 失败项不得阻塞主流程；必须记录已知问题。
- 不允许出现跨用户数据读取、未登录访问受保护数据、管理员权限绕过。

## 5. P0 Smoke 测试流程

### Flow-SMOKE-01：最小可演示闭环

目标：30 分钟内验证 MVP 是否可演示。

步骤：

1. 打开 `/register`，注册新用户 `creator-a-{timestamp}@postflow.local`。
2. 注册成功后进入 `/onboarding` 或 `/dashboard`。
3. 打开 `/accounts`，连接一个小红书账号。
4. 打开 `/studio/new`，输入选题：`夏季通勤穿搭：5件单品搞定一周`。
5. 选择小红书和公众号目标平台。
6. 点击 AI 生成或创建草稿，进入编辑页。
7. 在编辑页确认标题、正文、标签存在。
8. 点击快捷操作：重写、缩短、扩写、换标题，确认内容更新并产生版本。
9. 进入配图页，生成或选择 Mock 封面。
10. 进入适配页，生成小红书和公众号版本。
11. 进入预览页，确认小红书卡片和公众号预览可见。
12. 进入发布页，选择已连接账号，即时发布。
13. 等待任务从 `running` 变为 `succeeded`。
14. 打开 `/logs`，确认发布日志存在。
15. 打开 `/dashboard`，确认最近发布和 AI 额度变化。

预期结果：

- 用户主流程不中断。
- 发布任务最终成功。
- 刷新页面后草稿、版本、图片、适配版本、发布日志仍存在。

## 6. 鉴权与账号测试用例

### AUTH-001 注册成功

前置：未登录。

步骤：

1. 打开 `/register`。
2. 输入唯一邮箱和密码 `Postflow123`。
3. 输入确认密码。
4. 点击注册。

预期：

- 注册接口返回成功。
- 用户自动登录。
- 页面跳转到 Onboarding 或 Dashboard。
- 顶部/侧边栏显示当前用户信息。

### AUTH-002 弱密码注册失败

步骤：

1. 打开 `/register`。
2. 输入邮箱和密码 `short`。
3. 提交。

预期：

- 页面展示密码强度错误。
- 不创建用户。

### AUTH-003 重复邮箱注册失败

步骤：

1. 使用已注册邮箱再次注册。

预期：

- 页面展示“该邮箱已注册”或等价错误。
- 不覆盖原用户数据。

### AUTH-004 登录成功

步骤：

1. 打开 `/login`。
2. 输入已注册邮箱和正确密码。
3. 提交。

预期：

- 跳转 `/dashboard`。
- `/api/auth/session` 返回 user id、email、plan、isAdmin。

### AUTH-005 登录失败

步骤：

1. 输入正确邮箱和错误密码。

预期：

- 停留在登录页。
- 展示“邮箱或密码不正确”。

### AUTH-006 退出登录

步骤：

1. 登录后点击顶部退出按钮。

预期：

- session 清除。
- 跳转 `/login`。
- 访问 `/dashboard` 被重定向到登录页。

### AUTH-007 受保护页面拦截

步骤：

1. 未登录访问 `/dashboard`、`/studio/new`、`/admin`。

预期：

- 用户端受保护页跳转 `/login`。
- 管理端未登录也不能访问数据。

## 9. API 与安全测试用例

### API-001 未登录访问 v1 API

请求：

```bash
curl -i http://localhost:3000/api/v1/bootstrap
```

预期：

- 返回 401。
- 响应包含“请先登录”或等价错误。

### API-002 未登录访问 Admin API

请求：

```bash
curl -i http://localhost:3000/api/admin/overview
```

预期：

- 返回 401 或 403。

### API-003 普通用户访问 Admin API

步骤：

1. 普通用户登录。
2. 使用浏览器同 session 请求 `/api/admin/overview`。

预期：

- 返回 403。

### API-004 用户 A 读取用户 B 草稿

步骤：

1. 用户 A 创建草稿 A。
2. 用户 B 创建草稿 B。
3. 用户 B 尝试访问或修改草稿 A 的 API。

预期：

- 返回 404 或 403。
- 用户 A 草稿不被修改。

### API-005 用户 A 修改用户 B 发布任务

步骤：

1. 用户 A 创建一个 queued 发布任务。
2. 用户 B 尝试 PATCH `/api/v1/publish/{jobId}`。

预期：

- 返回 404 或 403。
- 任务状态不变。

### API-006 平台版本归属校验

步骤：

1. 用户 A 创建 draft + variant。
2. 用户 B 尝试 PATCH 用户 A 的 variant。

预期：

- 返回 404 或 403。
- 用户 A 数据不变。

## 10. 数据隔离测试流程

### Flow-ISO-01 双用户隔离

步骤：

1. 注册用户 A。
2. 用户 A 创建草稿、连接账号、生成图片、发布任务。
3. 退出。
4. 注册用户 B。
5. 打开 `/drafts`、`/accounts`、`/queue`、`/logs`。

预期：

- 用户 B 看不到用户 A 的草稿、账号、队列、日志。
- 用户 B 的 Dashboard 统计不包含用户 A 数据。

### Flow-ISO-02 切回用户 A

步骤：

1. 退出用户 B。
2. 登录用户 A。
3. 打开 `/drafts`、`/logs`。

预期：

- 用户 A 数据仍存在。
- 用户 B 数据不可见。

## 13. 错误与边界测试

### ERR-001 断网或 API 失败

步骤：

1. 在浏览器 DevTools 中模拟 offline。
2. 执行保存草稿或发布。

预期：

- 前端有错误提示。
- 页面不崩溃。
- 恢复网络后可重试。

### ERR-002 重复点击提交按钮

步骤：

1. 在注册、AI 生成、发布、支付确认按钮上快速连点。

预期：

- 不应创建大量重复数据。
- 按钮应有 loading 或禁用状态。
- 如果当前尚未完全防重，记录为 P1 改进项。

### ERR-003 长文本输入

步骤：

1. 在选题、正文、标签中输入超长文本。

预期：

- 页面不崩溃。
- 保存成功或出现明确限制提示。

### ERR-004 缺少平台账号时发布

步骤：

1. 新用户不连接账号。
2. 尝试发布。

预期：

- 发布页提示先连接账号。
- 不创建无效发布任务。

### ERR-005 AI 额度不足

步骤：

1. 将用户额度调低或消耗完。
2. 尝试 AI 生成。

预期：

- 返回 402 或页面展示额度不足。
- 引导到 `/billing`。

## 18. 已知风险与建议

| 风险 | 影响 | 建议 |
|------|------|------|
| Mock 发布不代表真实平台成功率 | 演示可用，上线风险高 | 后续接真实连接器前增加平台沙箱测试 |
| 图片 Data URL 写 SQLite | 大图片可能导致 DB 膨胀 | 生产前迁移对象存储 |
| 支付为 Mock | 无法验证真实扣款/回调 | 后续补订单状态机和支付回调测试 |
| 管理端权限模型简单 | 只适合单管理员 MVP | 后续补 RBAC 和审计日志覆盖 |
| Agent 视觉测试可能误判 | 文案/布局变化会影响定位 | 优先使用可见 label 和稳定业务路径 |
